# 코딩 작동 프롬프트 규칙 (Coding Prompt Rules)

> 이 문서는 SOWJS.KR 프로젝트의 AI 보조 코딩 시 따라야 할 핵심 원칙을 정의합니다.

---

## 1. 🧠 Think Before Coding (코드 타이핑 전에 생각부터)

코드를 작성하기 전에 반드시 먼저 생각하고 계획한다.

- **가정 명시**: 코드 작성 전 전제 조건(가정)을 명확히 서술한다.
- **불확실하면 질문**: 요구사항이 모호하거나 불명확할 경우, 임의로 구현하지 않고 먼저 질문한다.
- **혼란 시 멈춤**: 방향이 불분명하거나 충돌하는 요구사항이 있을 경우, 진행을 멈추고 명확화를 요청한다.

```
❌ 나쁜 예: 요구사항을 추측하여 바로 코드 작성
✅ 좋은 예: "이 함수가 null을 반환할 수 있다고 가정하고 진행합니다. 맞나요?"
```

---

## 2. ✂️ Simplicity First (단순하게 먼저)

요청된 것만 구현한다. 그 이상도, 그 이하도 아니다.

- **요청하지 않은 기능 추가 금지**: 미래를 위한 기능, 편의 기능을 임의로 추가하지 않는다.
- **불필요한 추상화 금지**: 당장 필요하지 않은 인터페이스, 추상 클래스, 레이어를 만들지 않는다.
- **과도한 에러 처리 금지**: 요청 범위를 벗어난 예외 처리나 방어 코드를 추가하지 않는다.

```
❌ 나쁜 예: 단순 함수 하나를 만들면서 팩토리 패턴, 전략 패턴까지 적용
✅ 좋은 예: 요청된 기능만 가장 단순한 방법으로 구현
```

---

## 3. 🔬 Surgical Changes (수술처럼 정밀하게)

변경은 최소한의 범위에서, 정확하게 이루어져야 한다.

- **요청된 부분만 수정**: 관련 없는 코드, 포맷, 변수명 등을 임의로 변경하지 않는다.
- **나머지는 건드리지 않기**: 기존 코드의 스타일, 구조, 로직을 존중하고 유지한다.
- **사이드 이펙트 최소화**: 변경이 다른 부분에 미치는 영향을 항상 인식하고 최소화한다.

```
❌ 나쁜 예: 버그 하나 고치면서 파일 전체를 리팩토링
✅ 좋은 예: 해당 버그가 발생하는 라인만 정확히 수정
```

---

## 4. 🎯 Goal-Driven Execution (목표 중심 실행)

막연한 작업 지시 대신, 구체적이고 검증 가능한 목표로 변환하여 실행한다.

- **"기능 추가"** → **"특정 테스트 케이스를 통과시키기"**
- **"성능 개선"** → **"API 응답 시간을 500ms 이하로 줄이기"**
- **"코드 정리"** → **"중복된 로직 X를 함수 Y로 추출하기"**

목표가 명확해야 완료 기준도 명확해진다.

```
❌ 나쁜 예: "팀 밸런싱 로직 개선해줘"
✅ 좋은 예: "10명 입력 시 두 팀의 예상 승률 차이가 5% 이내가 되도록 알고리즘 수정"
```

---

## 5. 📝 Git 커밋 메시지 컨벤션

협업과 이력 관리를 위해 아래의 커밋 메시지 규칙을 준수한다.

### feat: 새로운 기능 추가
### fix: 버그 수정
### docs: 문서의 수정
### style: 코드 수정 없이 스타일만 변경
### refactor: 코드 리펙토링
### test: 테스트 관련한 코드 추가/수정
### chore: 코드 수정 없이 설정 변경

---

## 적용 방법

이 규칙들은 AI에게 코딩 작업을 요청할 때 프롬프트 앞에 참조 문서로 첨부하거나,
개발 팀 내 코드 리뷰 기준으로 활용할 수 있습니다.

> **핵심 철학**: 좋은 코드는 많이 쓰는 것이 아니라, 정확하게 쓰는 것이다.
